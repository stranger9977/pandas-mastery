Level 1 · Explain like I'm 5

Sorting candy, growing sunflowers, and matching name tags

No jargon yet. Three ideas, told with kid stuff. Every fancy pandas trick in levels 2–4 is secretly one of these three moves.

First — what even is a table?

A DataFrame is a sticker chart

Each row is one thing that happened. Each column is one fact about it. And every row has a name tag on the left — pandas calls it the index. That name tag is the whole secret: pandas lines things up by name tag, not by position in line.

Idea 1 · GroupBy

Dump the bag. Sort into piles. Count each pile.

You dump your Halloween bag on the floor. Mom says: “how many reds do you have?” You don't stare at the messy heap — you sort into piles by color, then count each pile, then write one line per pile. That's it. That's groupby. Grown-ups call it split → apply → combine.

Watch the sort happen
Split: candies fly to their color pile. Apply: each pile gets counted. Combine: one answer per pile.
candy.groupby("color")["kind"].count()
# red      3
# green    2
# blue     1
Quiz

You sort your toys into bins by type, then find the heaviest toy in each bin. Which part is the “apply”?

Split = sorting into bins. Apply = the thing you do to each bin (max weight). Combine = writing down one answer per bin. Every groupby question decomposes into these three.

Idea 2 · Time series

The sunflower diary

Every morning you measure your sunflower and write the date next to the number. Congratulations — that's a time series: numbers with a clock attached, in order. Because time is attached, you can ask time questions: “how tall was it in June?” or “scoop each week's measurements into one jar — what's the weekly average?” Scooping days into weeks has a grown-up name: resample. It's just groupby where the piles are chunks of time.

Days get scooped into week jars
14 daily measurements → 2 week jars → 1 average per jar. resample = groupby over time buckets.
Quiz

You measured your sunflower every day, but you only want one number per week. That move is…

You keep all the information and summarize each time bucket (mean, max, last…). Deleting rows loses data; resampling condenses it. In pandas: flower.resample("W").mean().

Idea 3 · Merging

Matching kids to lunchboxes

Two lists: the class list (name → favorite animal) and the lunch list (name → lunch). To answer “what does the tiger kid eat?” you need one list. So you match rows by name. If Maya isn't on the lunch list, her lunch box is just… empty. Pandas writes NaN in the empty box instead of making something up.

Match by name tag
Same name → rows glue together. No match → the missing side becomes NaN.

And one more kid-sized idea that becomes a superpower later: “the last thing said still counts.” The ice-cream truck sign said $3 at 1:00. You buy at 1:17. Nobody re-announced the price at 1:17 — the price is whatever the sign last said before you bought. Matching “my moment” to “the most recent thing before it” is called an as-of match. Remember the ice-cream truck; it comes back at every level.

Quiz

Maya is on the class list but not the lunch list. After matching (keeping every kid in the class), Maya's lunch is…

A left join keeps every row from the left list and fills unmatched columns with NaN. Pandas never invents data on a merge — missing means missing.

Level 2 · Explain like I'm 14

A whole season of games

Same three ideas, now on data you'd actually keep: a season of basketball box scores. One row per game: date, team, opponent, pts_for, pts_against, win.

Time series

Dates are a superpower column

Read the file so pandas knows date is a real date, not a string that looks like one:

games = pd.read_csv("games.csv", parse_dates=["date"])

Now the .dt accessor unlocks calendar questions with zero math:

games["date"].dt.month          # 1..12
games["date"].dt.day_name()     # 'Friday'
games[games["date"].dt.month == 1]   # January games

And if you promote the date to the index (the name tags from Level 1), time becomes an address you can slice by:

g = games.set_index("date").sort_index()
g.loc["2026-01"]                # everything in Jan 2026 — partial string magic
g.loc["2026-01-10":"2026-02-01"]

Resample = the week jars from Level 1. With a datetime index, pandas can bucket rows by any calendar chunk:

g["pts_for"].resample("W").mean()    # avg points per week
g["pts_for"].resample("ME").sum()    # total points per month
Games → weekly buckets → one bar each
Each dot is a game's points. Fences drop at week boundaries; each bucket collapses to its mean. resample is groupby where the key is a time bucket.

Rolling = a sliding jar. Instead of disjoint weeks, drag a window of the last 5 games along the season — that's a team's form:

g["form5"] = g["pts_for"].rolling(5).mean()
The 5-game window slides
The amber window covers the last 5 dots; each step writes one smoothed value. Same split-apply-combine, but the “groups” overlap and slide.
Quiz

What's the real difference between resample("W") and rolling(7)?

resample collapses to one row per bucket. rolling keeps the original shape — every row gets a stat of its trailing window. It's the agg-vs-transform split, in time form. Also: rolling(7) = 7 rows; rolling("7D") = 7 days — not the same if games are unevenly spaced!

GroupBy

Per-team stats — and the fork in the road

The candy piles are now teams. One line answers “average points per team”:

games.groupby("team")["pts_for"].mean()

Want several stats at once with your own names? Named aggregation:

games.groupby("team").agg(
    gp    = ("pts_for", "count"),
    ppg   = ("pts_for", "mean"),
    wins  = ("win", "sum"),
)

Here's the fork every groupby question hits. Ask: do I want one row per team, or a value for every game?

  • .agg → collapse: one row per team (a leaderboard).
  • .transform → broadcast: the team's stat copied onto each of its games — same shape as the original, perfect for making new columns.
# each game vs the team's own season average
games["team_avg"] = games.groupby("team")["pts_for"].transform("mean")
games["hot"]      = games["pts_for"] > games["team_avg"]
agg collapses · transform broadcasts
Same split, same mean — the difference is the combine step: agg keeps one row per pile; transform sends the answer back to every original row.
Quiz

“Flag every game where a team scored 15+ over its own season average.” agg or transform?

The output is per-game (a flag on every row), so the group stat must be broadcast back: transform("mean"), then a vectorized comparison. If the deliverable were a per-team table, agg. Say the shape of the answer out loud and the verb picks itself.

Quiz

Quickest way to count games per team?

value_counts() is the one-liner for “how many rows per category”, sorted for free. transform would give you 720 repeated numbers; the division only works if every team played the same schedule.

Merging

Joining the season to everything else

Team info lives in a second table — teams: team, coach, arena. Exact merge, matching by key:

full = games.merge(teams, on="team", how="left")
  • how="left" — keep every game; a team missing from teams gets NaN coach (Maya's lunchbox).
  • how="inner" — keep only matches (games can silently disappear!).
  • how="outer" — keep everything from both sides.

Now the ice-cream truck grows up. Power rankings drop every Monday. What was each team's ranking on game day? A Thursday game shouldn't match Thursday's ranking (there isn't one) — it should match the most recent Monday's. Exact merge can't do that. merge_asof exists exactly for this:

games = games.sort_values("date")
ranks = ranks.sort_values("date")     # both MUST be sorted by the time key

with_rank = pd.merge_asof(
    games, ranks,
    on="date",            # the time key
    by="team",            # match within team first
    direction="backward", # most recent ranking AT OR BEFORE game day
)
Each game reaches back to the latest ranking
Squares = games, dots = Monday rankings. Each game scans backward and grabs the most recent ranking at-or-before it. That's direction="backward" — the default, and almost always what you want.
Quiz

Why would direction="forward" be cheating for a prediction model?

Forward grabs the next ranking — which already knows the game's result. That's data leakage: your model looks brilliant in backtest and useless live. “backward” = only information that existed at that moment. This one idea is worth saying out loud in any interview.

Level 3 · Explain like I do this for work

Trades and quotes, on the clock

Now the data is market microstructure: a trades table (ts, symbol, price, size) and a quotes table (ts, symbol, bid, ask). Everything from Levels 1–2 applies — the stakes and the argument lists grow.

Time series

The DatetimeIndex, properly

trades = pd.read_csv(
    "trades.csv",
    parse_dates=["ts"],              # strings → Timestamps at read time
    dtype={"symbol": "category"},    # repeated strings → tiny ints
).sort_values(["symbol", "ts"])

Promote time to the index and you get addressing, slicing, and clock filters:

t = trades.set_index("ts").sort_index()
t.loc["2026-06-22 09:30":"2026-06-22 10:00"]   # a slice of the morning
t.between_time("09:30", "16:00")               # regular hours, any day

Downsampling ticks into bars. Nobody models raw ticks straight away — you build bars. ohlc() is a ready-made aggregation: first/max/min/last of each bucket:

bars = t.groupby("symbol", observed=True)["price"].resample("5min").ohlc()
#                                  open    high     low   close
Ticks → one 5-minute OHLC bar
The bucket keeps four survivors: first tick (open), highest (high), lowest (low), last (close). Everything else is summarized away — that's downsampling with intent.

Features over time — and the leakage rule. Rolling and EWM windows end at the current row, so “as of now” includes now. If the feature will predict something at time t, shift it:

bars["vol20"]  = bars["close"].pct_change().rolling(20).std()
bars["ewm10"]  = bars["close"].ewm(span=10).mean()
bars["feat"]   = bars["vol20"].shift(1)    # only yesterday's info predicts today
All three lines keep the row count — rolling/ewm/shift are transform-shaped verbs. So yes: each lands as its own column, one value per bar (the first 19 are NaN while the 20-window fills). You never choose to broadcast; the verb already has that shape. Reading the vol20 line inside-out: pct_change() = returns per bar → rolling(20) = trailing window of 20 bars → .std() = how bumpy — i.e. volatility.
Chant it: shift · backward · split. shift(1) your rolling features; merge_asof direction="backward"; split train/test by time. Three doors leakage sneaks through, three locks.
Quiz

rolling(20) vs rolling("20min") on tick data — what's the trap?

Tick arrivals are irregular — row-count windows stretch and shrink in wall-clock terms. Time-based windows (rolling("20min"), needs a datetime index) mean what they say. Pick the one that matches the sentence you'd say about the feature.

GroupBy

Two keys, two verbs, one Grouper

Per-symbol VWAP — a collapse (agg) built from vectorized pieces:

trades["notional"] = trades["price"] * trades["size"]
vwap = (trades.groupby("symbol", observed=True)
              .agg(notional=("notional","sum"), volume=("size","sum")))
vwap["vwap"] = vwap["notional"] / vwap["volume"]

Per-symbol z-score of size — a broadcast (transform), because the answer belongs on every row:

g = trades.groupby("symbol", observed=True)["size"]
trades["size_z"] = (trades["size"] - g.transform("mean")) / g.transform("std")
Why transform: the z-score round trip
Rows split into symbol buckets → each bucket computes its mean & std → those numbers ride BACK to every original row → each row asks "how unusual am I, in units of my own symbol's typical?" A 300-lot is huge for AAA and boring for BBB — that's why the stats must be per-group and per-row at once.

Group by a category AND a time bucket at once — this is the underrated tool. pd.Grouper lets a time bucket sit inside a normal groupby, no index gymnastics:

bars = trades.groupby(
    ["symbol", pd.Grouper(key="ts", freq="5min")], observed=True
).agg(open=("price","first"), high=("price","max"),
      low=("price","min"),   close=("price","last"),
      volume=("size","sum"))
That one pattern — groupby([category, pd.Grouper(key=time, freq=...)]) with a named agg — builds almost any bar table in one pass. It's the workhorse of every market-data exercise.
Quiz

Why does VWAP have to be sum(price×size)/sum(size) rather than a plain mean() of price?

VWAP is a weighted mean, weights = size. Compute the pieces vectorized (notional column, two sums), divide once. Bonus habit: doing it via two .agg sums stays fast; a .apply with a lambda per group does not.

Quiz

You need each trade tagged with its symbol's total volume that day (to compute participation %). Verb?

agg-then-merge works but is two steps and a join; transform is the built-in one-liner for exactly this shape. Save apply for when no built-in fits — it's a Python loop in disguise.

Merging

The trade meets its quote

Exact merges get defensive at this level — say what you expect, make pandas check it:

full = trades.merge(symbols_meta, on="symbol",
                    how="left",
                    validate="m:1",        # many trades → one meta row; raise if not
                    indicator=True)        # adds _merge: both / left_only
full["_merge"].value_counts()              # audit the join before trusting it

Now the signature market-data move. A trade prints at 10:15:03.2 — what were the bid and ask at that instant? Quotes don't update on a schedule; the prevailing quote is the ice-cream sign: whatever was last posted. This is inexact, as-of matching:

trades = trades.sort_values("ts")
quotes = quotes.sort_values("ts")          # sorted by the ON key — required

taq = pd.merge_asof(
    trades, quotes,
    on="ts",                     # inexact time key
    by="symbol",                 # exact match on symbol FIRST
    direction="backward",        # most recent quote at-or-before the trade
    tolerance=pd.Timedelta("5s"),# refuse stale quotes → NaN instead of lies
)
taq["spread"] = taq["ask"] - taq["bid"]
taq["mid"]    = (taq["bid"] + taq["ask"]) / 2
merge_asof with by="symbol" — two lanes, no crossing
Amber squares = trades, teal dots = quotes, one lane per symbol. Each trade reaches back to the latest quote in ITS OWN lane. A trade further back than the tolerance gets NaN — honest, not wrong.
Quiz

merge_asof raises ValueError: left keys must be sorted. What's the actual fix?

There is no sort flag — the algorithm walks both tables once in time order (that's why it's fast), so it demands sorted inputs. NaNs in the key break sortedness too. The rhythm: sort both → merge_asof → audit the NaNs.

Level 4 · Expert

Ninety million rows walk into a bar

Same three ideas. But now it's every tick for 500 symbols, the naive version doesn't fit in RAM, and being right includes being fast, small, and audited. This level is about handling heavy data with care — and downsampling it on purpose.

Discipline 1 · Load small, not everything

Memory is won at read time

df = pd.read_parquet(
    "ticks.parquet",
    columns=["ts","symbol","price","size"],          # never load what you won't use
    filters=[("symbol","in",universe),
             ("ts",">=", start)],                    # predicate pushdown: skip row groups on disk
)
df["symbol"] = df["symbol"].astype("category")       # 500 strings → int codes
df["size"]   = df["size"].astype("int32")            # halve the ints
df.info(memory_usage="deep")                         # always know your footprint
  • category for any column with few unique strings — often a 10–20× shrink.
  • float32 / int32 where precision allows — half the bytes, double the cache hits.
  • Parquet over CSV: typed, compressed, column-selectable, filterable before RAM.
  • Still too big? Chunk it (read_csv(..., chunksize=1_000_000)): aggregate each chunk, combine the partials — a hand-rolled map-reduce. The trick: keep per-chunk sums and counts, never per-chunk means (means of means are wrong when chunk sizes differ).
Quiz

A 40 GB tick file, you need daily volume for 3 of 500 symbols. Best first move?

The cheapest row is the one you never load. Predicate pushdown means the parquet reader skips whole row groups on disk — you touch ~0.6% of the data. “Filter after loading” already OOM'd. Naming this move calmly is exactly what a “my machine crashes” interview question wants.

Discipline 2 · Downsample with intent

Resample is a scalpel, not a blender

At scale you choose the coarsest granularity that still answers the question — and you control the bucket edges, because off-by-one-bucket bugs are silent:

bars = (df.set_index("ts")
          .groupby("symbol", observed=True)["price"]
          .resample("15min",
                    label="right",    # stamp the bar with its END time...
                    closed="right")   # ...and own the boundary tick
          .ohlc())
  • label/closed: does the 09:30–09:45 bar get stamped 09:30 or 09:45, and which edge owns a tick at exactly 09:45? Financial convention is usually right/right — a bar is named for when it completes. Mislabeling shifts every feature one bar into the future: silent leakage.
  • origin="start_day" anchors buckets to midnight so 15-min bars land on :00/:15/:30/:45 regardless of when trading started.
  • Upsampling is the reverse trap: resample("1s").ffill() manufactures rows. Fine for alignment, but never compute “average price” from ffilled rows — you're averaging your own echo.
  • One pass, many stats: give resample/groupby a named agg instead of calling it five times. Each built-in agg is a fused, C-speed reduction; five separate calls = five scans.
Quiz

Your 15-min bars use label="left". Feature = bar close, target = next bar's return. What just happened?

Labels decide what “as of 09:30” means downstream — in the merge_asof, in the train/test split, everywhere. Convention: label the bar with its close time (label="right", closed="right"), then a backward asof can never see inside an unfinished bar.

Discipline 3 · GroupBy at scale

Stay on the C path

g = df.groupby("symbol", observed=True, sort=False)

daily = df.groupby(["symbol", df["ts"].dt.normalize()],
                   observed=True).agg(vol=("size","sum"))   # built-in agg: C speed

df["ret"] = g["price"].transform("pct_change")              # per-group, no loop
  • observed=True on categorical keys — otherwise pandas materializes every possible category combination: 500 symbols × 500 dates of empty rows. (It's the default in pandas 3.x, but say it; it shows you've been burned.)
  • sort=False skips sorting group keys you don't need sorted.
  • The performance ladder: vectorized column math > built-in agg/transform ("mean") > apply(lambda). apply calls Python once per group — 500 groups is fine, 5 million groups is a coffee break. Before writing apply, ask: can I compute pieces as columns first (the VWAP trick), then agg the pieces?
  • Repeated groupbys on the same key? Build g once and reuse it.
Quiz

Per-symbol daily VWAP on 90M rows. Which shape stays fast?

Same math, ~100× gap. The apply/lambda runs Python per group; the loop rescans the frame per symbol. Compute-pieces-then-agg keeps every step a single C-speed pass. This exact question is the classic “make it faster” follow-up.

Discipline 4 · merge_asof at scale

Why sorted, really — the two-pointer walk

merge_asof isn't “a merge with vibes.” It's a single simultaneous walk down two sorted tables — one pointer each, never backing up. That's O(n+m) for any size, and it's exactly why unsorted input is a hard error, not a warning:

One pass, two pointers, zero lookups
The quote pointer only ever moves right. Each trade takes the last quote the pointer passed. 90M trades × 90M quotes = one linear scan, not a search per trade — but only because both are sorted.

The production checklist around it:

for df_ in (trades, quotes):
    assert df_["ts"].is_monotonic_increasing, "sort before asof"
    assert df_["ts"].notna().all(),           "NaT breaks ordering"

taq = pd.merge_asof(trades, quotes, on="ts", by="symbol",
                    direction="backward",
                    tolerance=pd.Timedelta("5s"),
                    allow_exact_matches=False)   # quote stamped AT the trade's
                                                 # instant: did it exist "before"?
taq["bid"].isna().mean()                         # audit: % refused as stale
  • tolerance converts “stale but technically previous” into honest NaN. Always report the refusal rate.
  • allow_exact_matches=False is the paranoid leakage setting: a quote with the identical timestamp may not have been visible when the trade fired. Know the flag, mention the ambiguity.
  • Duplicate timestamps on the right side? The last one wins — dedupe deliberately (quotes.drop_duplicates(["symbol","ts"], keep="last")), don't let it happen to you.
  • Exact merges at scale: validate="m:1" is a safety net against the many-to-many explosion — a dup key on both sides multiplies rows and eats RAM before you notice. Filter (rows AND columns) before joining, not after.
Quiz

After an asof join, 38% of trades have NaN quotes. First reaction?

A huge refusal rate is the join telling you something — e.g. symbol "AAPL" vs "aapl ", tz-aware vs naive timestamps, or a vendor gap. Papering over it with ffill/looser tolerance bakes a lie into every downstream number. Diagnose, quantify, then decide.

Toolkit · Market vocab

The words, without the mystique

Every term you'll meet in the exercises, in plain English with a sports hook. Tap a card to flip it; mark know it when it sticks. The pandas is easy once the words stop costing you brainpower.

The scalper — prices

One image carries half the glossary: a ticket scalper outside the stadium. He'll buy your ticket for $80 and sell you one for $100, all day, adjusting his numbers as tip-off approaches.

bid know it
The price buyers will pay right now. The scalper's "I'll give you $80 for yours." Always the lower number.
ask (offer) know it
The price sellers will accept right now. "Yours for $100." Always above the bid.
spread know it
ask − bid. The scalper's $20 gap — his paycheck for standing there taking risk. Tight spread = busy, competitive market.
mid know it
(bid + ask) / 2. The fair-ish price between the two advertisements. The usual "true price" proxy in analysis.
quote know it
The current advertised pair — bid & ask (with sizes). It's an advertisement, not a transaction; it changes constantly even when nobody trades.
crossed quote know it
bid > ask — someone's willing to pay more than sellers are asking. Almost always bad data; a cleaning flag, like a negative game score.
prevailing quote know it
The quote in force at a given instant = the latest one at-or-before it. The ice-cream sign. This is exactly what merge_asof fetches.
liquidity know it
How much you can trade without moving the price. "Providing liquidity" = being the scalper: quoting both sides so others can always deal.
Trades & bars — events
trade / print know it
An actual transaction — someone accepted the advertised price. Has a price and a size. "It printed at 99.10" = a trade happened there.
tick know it
One row of raw market data (a single trade or quote update). "Tick data" = every event, unaggregated — the play-by-play log.
size / volume know it
size = shares in ONE trade. volume = total shares over a period. Like one play's yards vs the game's total yards.
notional know it
price × size = the dollars that changed hands. 200 shares at $50 = $10,000 notional. The money-weighted view of a trade.
bar / OHLC know it
A time bucket's four-number summary: Open (first), High (max), Low (min), Close (last). A box score for 5 minutes of trading. .resample("5min").ohlc()
VWAP know it
Volume-Weighted Average Price = sum(price×size) / sum(size). The average price actually paid, where a 10,000-share trade counts 100× a 100-share one. The benchmark for "did I trade well today."
Stats on prices — features
return know it
The %-change bar to bar: pct_change(). Log return = np.log(p).diff() — nearly identical for small moves, and adds up cleanly across periods. Prices aren't comparable across stocks; returns are.
volatility (realized) know it
The std of returns over a window: how bumpy the ride has been. ret.rolling(20).std(). Annualize by ×√(periods per year). A team analogy: not their average margin, but how wildly it swings.
z-score know it
(x − mean) / std — "how unusual is this, in units of typical?" z=+2 ≈ top 2%. Per-group z-score (via two transforms) asks: unusual for its own symbol.
EWM / span know it
A moving average where recent points count more, fading exponentially. ewm(span=10).mean() — span ≈ how many periods really matter. Small span = fast & jumpy; big = slow & smooth. A team's "form" where last night matters most.
stale know it
Old information still on the books — a quote from 5 minutes ago in a fast market. tolerance= in merge_asof is your staleness bouncer: too old → NaN instead of a lie.

Bid = what he pays (low), ask = what he charges (high), spread = ask − bid = his compensation for carrying risk. If bid ever exceeds ask, the quote is "crossed" — in data, that's a cleaning flag.

Vocab quiz

Inside-out: pct_change = returns; rolling(20) = the trailing window; std = the bumpiness. And because rolling is transform-shaped, it lands as its own column — no broadcasting decision needed.

Vocab quiz

ewm(span=5) vs ewm(span=50) on the same series — which statement is right?

Span sets how far back the "memory" meaningfully reaches. Small span = recent games dominate (hot-hand detector); large span = season-long form. Both produce one value per row.

Toolkit · Day-of plotting

Five plots you can type with your eyes closed

In the exercise, plots aren't decoration — they're how you narrate. Each plot answers exactly one question, and you say the question out loud as you type it. Memorize five shapes, total.

The five
import matplotlib.pyplot as plt

# 1 THE LOOK — "what does this series even look like?"
df["close"].plot(title="AAA close")

# 2 THE DISTRIBUTION — "are there outliers / how heavy are the tails?"
df["ret"].hist(bins=50)
plt.title("1-min returns")

# 3 RAW vs SMOOTH — "is my cleaning/smoothing doing what I claim?"
ax = df["close"].plot(alpha=.5, label="raw")
df["close"].rolling(20).mean().plot(ax=ax, label="20-bar mean")
ax.legend()

# 4 THE HOLES — "where is data missing, and is it random?"
df.isna().sum().plot.bar(title="missing by column")

# 5 PRED vs ACTUAL — "is the model biased, or just noisy?"
plt.scatter(y_true, y_pred, alpha=.4)
plt.axline((0, 0), slope=1)          # the perfect-model line
plt.xlabel("actual"); plt.ylabel("predicted")
  • Title everything. An untitled plot forces the interviewer to ask what they're looking at — that's negative narration.
  • One plot, one sentence. Before typing, say the question; after it renders, say the answer. "Here's the raw series — those ×2 spikes look like bad prints; I'll flag |log return| > 0.1."
  • In a notebook, plots render on cell-run. If nothing shows in a script: plt.show().
  • Groupwise version of #1 when comparing symbols: df.groupby("symbol")["close"].plot(legend=True) — or better, pivot to columns first and .plot() the frame.
